## 架構總覽（四層）

```
使用者
  ↓（現階段直接溝通，未來由秘書 Agent 接手）
PM Agent（整合需求 · 壓縮 context · 派工）
  ↓ Sub Agent 模式
各領域 Agent（文案 / 程式 / 美術）
  ↓ Agent Team 模式
各角色（執行 / QA / 優化）
```

## 已確認決策

### 2026-04-30

**決策 1：混合派工模式**
- PM → 各領域：Sub Agent（各自獨立，互不干擾）
- 領域內部：Agent Team（角色可直接溝通，領域 Agent 擔任 manager）

**決策 2：QA 閉環 = SOP 機構記憶**
- 優化角色不即時修正當次任務
- 將問題根因寫進角色 SOP 文件的「已知易錯點」欄位
- 後代角色啟動時自動繼承歷史教訓

```
執行 → QA → 優化 → 更新 SOP 文件 → 下次啟動繼承
```

### 2026-05-03

**決策 3：角色文件骨架（七個 Section）**

所有角色文件統一繼承此結構，覆蓋優先級為 `角色文件 > 領域文件 > 全局文件`：

| Section | 名稱 | 必填 |
|---------|------|------|
| 一 | 角色定位 | ✅ |
| 二 | 主要任務 | ✅ |
| 三 | 工作流程（3.1 輸入規格 / 3.2 執行步驟 / 3.3 輸出規格） | ✅ |
| 四 | 專家知識 | ✅ |
| 五 | 檢驗標準 | ✅ |
| 六 | 已知易錯點 | ✅ |
| 七 | 參照 References | ⬜ 選填 |

三種內容注入點：`專家的知識` → §四、`檢驗的標準` → §五、`已固定的工作流程` → §三 3.2

**決策 4：五種資料契約（§4 全局資料傳遞規則）**

| 契約 | 方向 | 時機 |
|------|------|------|
| 派工單 | PM → 領域 | PM 指派任務 |
| 交付包 | 領域 → PM | 任務完成或中斷 |
| 釐清請求 | 下游 → 上游 | 需求不明 / 矛盾 / 超出範疇 |
| 上下文壓縮包 | PM → 下一個 PM | 見觸發規則 |
| 錯誤回報單 | QA / 執行 → PM / 優化 | QA 未通過 / 執行失敗 |

**決策 5：上下文壓縮觸發（方案 C，混合觸發）**
- 主觸發（自然斷點）：派工前、收齊交付包後、最終審核前
- 備援觸發（Token 閾值）：使用率達 70% 強制壓縮，標記 `觸發原因: Token 閾值`
- 選擇理由：斷點讓人工介入有清晰時機；備援防止意外爆 context

**決策 6：PM Agent 職能拆分為 5 個 Skills / Prompts**

| 編號 | 職能 |
|------|------|
| 1 | 輸入摘要化與重點整合 |
| 2 | 讀取領域 GuideLine，生成功能規格書 |
| 3 | 角色指派與創建（依照角色架構範本） |
| 4 | 上下文壓縮與快照產出 |
| 5 | 最終審核與錯誤回報 |

## 文件繼承架構（OOP 概念）

```
全局文件（Base）
├── 領域文件（繼承全局）
│   └── 角色 SOP 文件（繼承領域）
```

| 文件 | 讀者 | 狀態 |
|------|------|------|
| 全局文件 | 所有 Agent | ✅ v0.2 完成（§2 輸出格式 + §4 資料傳遞規則 均已填入） |
| 領域文件 | 該領域所有角色 | ⏸ 下一步：先從文案領域開始 |
| 角色 SOP 文件 | 該角色本身 | ⏸ 待領域文件完成後設計 |
