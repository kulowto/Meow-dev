## 文件用途與閱讀說明

此文件記錄每個 Agent 角色的**設計思路與職責背後的原因**，而非操作細節。

目的是讓未來在擴充新角色、或需要靈活變通時，能理解系統設計的脈絡，做出更合理的決策，而不只是照著 SOP 執行。只知道「怎麼做」，遇到新情境時會很難判斷；理解「為什麼這樣做」，才能在邊界情況下做出正確的取捨。

## PM Agent

### 職責摘要

PM Agent 是整個系統的**核心樞紐**，負責接收使用者（或秘書 Agent）的輸入，拆解需求後指派給各領域，並在任務完成前持續追蹤進度與品質。

### 設計背後的思路

**為什麼 PM 是最複雜的角色？**
PM 的對接端口最多：上接使用者 / 秘書，下接三個領域（文案 / 程式 / 美術），還需要在領域之間協調依賴關係。每一個職能大概率需要獨立的 Skill 或 Prompt 來維護，因為把所有邏輯塞進一個角色會讓 context 失控。

**職能清單與設計理由**

| 職能 | 為什麼需要這個 |
|------|--------------|
| 輸入摘要化 | 使用者或秘書傳來的資料格式不固定，需要先標準化才能派工 |
| 生成領域規格書 | 讓各領域 Agent 有明確的需求依據，而不是各自解讀原始輸入 |
| 角色指派與創建 | PM 要知道每個任務該交給誰，並能依照角色架構範本生成對應角色 |
| 上下文壓縮（方案 C） | PM 的任務時間跨度難以預測，純靠斷點有爆 context 風險；純靠 Token 閾值又容易中途切斷執行邏輯，因此採混合方案 |
| 最終審核 | 確保交付物符合硬性規範，不把問題留給使用者發現 |
| 跨領域依賴管理 | 部分任務有前後依賴（例如文案需先完成，美術才能配圖），PM 需要判斷並行或串行 |
| 需求變更處理 | 使用者中途改需求是常態，PM 需要評估影響範圍並通知相關領域 |
| 整體進度追蹤 | 壓縮包和最終審核都需要這份進度狀態，是 PM 的核心資訊來源 |

**上下文壓縮的觸發設計（方案 C，混合觸發）**

選擇理由：自然斷點讓人工介入有清晰的時機點，可以在斷點審視進度並做決策；備援觸發則防止意外爆 context。

- **主觸發（自然斷點）**：派工前、收齊各領域交付包後、最終審核完成前
- **備援觸發（Token 閾值）**：使用率達 70% 時強制壓縮，壓縮包標記 `觸發原因: Token 閾值`

**PM Agent 預計拆分的 Skills / Prompts**

每個職能獨立維護，避免單一角色過於肥大：

| 編號 | Skill / Prompt 用途 |
|------|-------------------|
| 1 | 輸入摘要化與重點整合 |
| 2 | 讀取領域 GuideLine，生成功能規格書 |
| 3 | 角色指派與創建（依照角色架構範本） |
| 4 | 上下文壓縮與快照產出 |
| 5 | 最終審核與錯誤回報 |
