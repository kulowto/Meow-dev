# 影片轉工具開發流程規範 v0.3

適用範圍：以教學影片為來源，轉換為可開發的工具規格或自動化腳本的任務
工具依賴：MEMO.AI（文字轉錄）+ AI 解析
不限軟體：PS / AI / 任何有操作步驟的教學影片皆適用

> **設計理念參考**：本規範是整個工作流程的「粗濾網」。
> 完整的濾網模型說明（為什麼分兩層、上下游如何對接）見：
> `(AI_Read) 開發理念_流程設計與資訊濾網模型.md`

---

## 概述

本規範定義一套標準化流程，讓 AI 從短影片教學中自動萃取操作步驟、轉換為工具開發規格、並實作為可執行的自動化工具。

核心概念：
- **影片** 是需求來源，不是文件
- **MEMO.AI** 是人工字幕的替代品，解決 AI 無法直接「看影片」的問題
- **AI 解析** 負責修正語音辨識誤差、結構化步驟（粗濾網）
- **使用者確認** 是品質把關的唯一節點
- **Debug 紀錄** 用於 Phase 3 二次優化（細濾網），不在初步處理時使用

---

## 工作流程（5 個 Phase）

### Phase 1｜影片來源確認

輸入：YouTube 連結 或 本機影片檔案

AI 執行：
1. 開啟 YouTube 頁面截圖（Playwright）→ 識別軟體介面（PS / AI / 其他）
2. 讀取頁面 metadata → 取得標題、時長
3. 截圖確認第一幀畫面內容與燒入字幕

輸出：影片資料卡
```
軟體：PS / AI / 其他
標題：
時長：
初步判斷：是否適合製作工具（見篩選規則）
```

---

### Phase 2｜MEMO.AI 文字轉錄

**現行流程（AI 透過 CDP 自動操作）**：
1. 以 CDP 模式啟動 MEMO.AI：
   `Start-Process "C:\Users\{username}\AppData\Local\Programs\Memo\Memo.exe" -ArgumentList "--remote-debugging-port=9222"`
2. CDP 連線：`http://localhost:9222/json` 取得 WebSocket URL
3. 填入 YouTube URL → 點擊第一個「確認轉錄」按鈕
4. **等待對話框出現後，依序設定以下三項（必填，不可略過）**：
   - 模型：**高品質 - Medium**（下拉選單，選錯會影響辨識品質）
   - 運算方式：**GPU 加速**（勾選，有顯卡時速度較快）
   - **自動斷句**：勾選（影響逐字稿分段品質）
5. 點擊對話框內的「確認轉錄」→ 等待轉錄完成
6. 切換至「語音轉文字」tab → 提取逐字稿文字 → 存 `.txt`

**⚠️ CDP 腳本對話框設定注意事項**：
對話框內的模型選單和勾選項必須在 `DIALOG_CONFIRMED` 前完成設定。
相關腳本：`C:/Users/AkatsukiNeko/AppData/Local/Temp/memo_batch.py`

**重要限制**：
- MEMO.AI **僅能轉錄語音**，無法擷取影片中的純文字畫面
- 若影片無口述語音（全靠字幕文字呈現步驟），需改用其他工具或使用者手動補充
- 需改用工具時，在工具規格草稿的來源欄位標注「無語音，需手動補充」

輸出：原始逐字稿（包含語音辨識錯誤，此為正常現象，Phase 3 修正）

---

### Phase 3｜AI 步驟萃取與清洗

輸入：原始逐字稿

AI 依序執行以下六項解析：

---

**1. 語音辨識難處理影片標注**

在開始清洗前，先判斷此影片是否屬於「語音辨識難處理型」。若符合以下任一條件，在規格草稿頂部加特別說明：
- 影片操作大量依賴視覺畫面（面板數值、形狀選擇等），語音幾乎不說出具體數值
- 講者有明顯口音，造成系統性誤辨

標注格式：
```
> ⚠️ **語音辨識特別說明**：[說明原因]。Phase 3 步驟萃取需搭配影片畫面人工補充，
> 純靠逐字稿無法取得完整參數。
```

---

**2. 語音辨識修正**

對照常見模式（見附錄指引）修正誤辨字。
無法判斷時標記 `[辨識存疑]` 並保留原文，等 Phase 4 使用者確認。

修正紀錄格式（每份規格草稿必須包含）：
```markdown
## Phase 3：語音辨識修正紀錄（Debug 保留）

| 原始辨識 | 修正為 | 備註 |
|---------|--------|------|
| 原始錯誤文字 | 正確術語 | 誤辨原因（音近/連讀/口音等）|
```

---

**3. 過濾非操作性內容**

- 刪除：開場白、閒話、CTA（記得訂閱、點讚等）、語氣詞
- 刪除：「数值请看VCR」類指引觀眾的句子（不含操作資訊）
- 保留：含動詞的操作句（「打開」「選取」「添加」「設定」「拉」「拖」「複製」「刪除」）

---

**4. 步驟結構化**

每個步驟的標準格式：

```
步驟 N：[自動化標注] [操作動詞] [目標] [參數或方向（若有）]
  ├─ [自動化] ✅ 全自動 / ⚠️ 半自動 / ❌ 手動介入 / ✅ 告知邏輯
  ├─ [原始辨識] 「對應原始語音辨識文字」
  ├─ [修正] 修正說明（若有）
  ├─ ⏸ 手動介入說明（若需使用者操作）
  └─ 📝 建議記錄說明（若數值待累積）
```

**步驟 0（前置確認步驟）— 必填**：
每份工具規格草稿的第一個步驟必須是前置確認。
- 用途：確認輸入素材是否就緒，或引導使用者選擇操作目標
- 判斷邏輯：若已滿足條件（如已有開啟的圖片），可在工具內自動跳過告知；若未滿足，告知使用者應該準備什麼
- 範例：「請先開啟一張照片」/ 「請在圖層面板中點選要套用效果的文字圖層」

**最後步驟（完成告知步驟）— 必填**：
每份工具規格草稿的最後一個步驟必須是完成告知。
- 格式：「[效果名稱] 已完成！[後續建議操作（若有）]」
- 若有使用者需要自行完成的項目（輸入文字、點綴元素等），同步告知

---

**5. 主觀選擇參數處理**

當步驟中有主觀性的選擇（顏色、漸變、形狀等），遵循以下規則：
1. 在步驟描述中加入「優先詢問使用者」的邏輯
2. 同時提供預設值（含具體數值，如色碼）
3. 在可調參數表格中標注「優先詢問使用者」

---

**6. 效果原理標注**

若影片說明技術原理（如「減藍 = 加黃」），單獨記錄為「效果原理」欄位。

---

輸出：結構化步驟草稿（等使用者確認）

---

### Phase 4｜使用者確認 + 規格定稿

**使用者確認清單**：
- [ ] 步驟順序是否正確
- [ ] 有無遺漏的步驟
- [ ] 辨識存疑項目是否已確認
- [ ] 可調整參數的預設值與範圍是否合理
- [ ] 工具用途說明是否準確

**Phase 4 修正紀錄格式**：
若 Phase 4 有修正，在規格草稿中新增以下表格：
```markdown
## Phase 4：使用者修正紀錄

| 步驟 | 原草稿描述 | 修正後 | 說明 |
|------|---------|--------|------|
```

**可調參數表格格式規範**：
可調參數表格必須包含以下欄位（不可省略「作用說明」和「固定/可調」）：

```markdown
| 參數名稱 | 預設值 | 作用說明 | 固定/可調 |
|---------|--------|---------|---------|
```

對於「數值不固定」的可調參數：
- **作用說明**欄必須說明此參數調高或調低分別會有什麼視覺效果（讓使用者調整時能立即理解）
- 步驟描述中，針對可調參數加入「[使用者調整提示]」區塊，說明何時需要調整、調整方向

輸出：定稿步驟清單 + 可調參數表

## 篩選規則

### 適合製作成工具的

- 有明確步驟序列（動詞性操作，順序固定）
- 參數有具體數值，或可設合理預設值
- 效果可程式化重現，不依賴「靠感覺調整」
- 軟體為 PS（目前唯一有 Python COM 支援的工具）

### 暫時跳過的

| 情況 | 原因 |
|------|------|
| AI（Illustrator）影片 | 無 Python COM 支援 |
| 純手感操作（「拉到你滿意為止」） | 無法程式化 |
| 過於依賴特定素材的步驟 | 無法泛化 |
| 逐字稿還原不出完整步驟 | 資訊不足，需補充 |

## 附錄：語音辨識修正參考

Phase 3 處理時，對照以下常見規律進行修正。
已知的具體錯誤對照表（細濾網）：`(AI_Read) 語音辨識錯誤紀錄_Debug.md`

**常見誤辨規律（粗濾網層級）**：
1. **快捷鍵字母**：「Ctrl+勾」= Ctrl+J；「三件+字母」= Ctrl+Shift+Alt+字母
2. **PS 術語音近**：遇到不通順的詞，先查 Debug 紀錄中的「軟體術語誤辨」分類
3. **非操作句**：看VCR、訂閱、語氣詞 → 過濾
4. **連讀誤辨**：兩個相鄰的術語名稱被連讀成一個詞（如「深度打小」= 深度、大小）
5. **兩岸術語**：影片為簡體術語，輸出請換算為繁體 PS 名稱
