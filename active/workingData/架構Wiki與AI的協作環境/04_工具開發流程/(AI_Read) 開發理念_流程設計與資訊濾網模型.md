# 開發理念：流程設計與資訊濾網模型

> **這份文件是給誰看的**：
> 如果你是新接手這個工作流程的 AI，不確定某個決策背後的邏輯，或不知道為什麼某些文件要分開放，先看這份。

## 核心概念：資訊濾網模型

### 什麼是濾網模型？

把資訊處理過程想像成廚房裡的過濾流程：
- 原始材料（影片教學）帶有大量雜訊（語音辨識錯誤、非操作內容、口頭語）
- 我們用「濾網」把雜訊過濾掉，得到乾淨的結構化資訊
- 雜訊太多時，一個濾網不夠，我們用**兩層濾網**依序處理

### 兩層濾網的定義

| 濾網 | 口徑 | 文件 | 特性 |
|------|------|------|------|
| 粗濾網 | 大（通用規則） | `影片轉工具開發流程規範.md` | 快速過濾大量雜訊；通用規則；允許少量誤差殘留 |
| 細濾網 | 小（針對性修正） | `語音辨識錯誤紀錄_Debug.md` | 針對已知錯誤模式；逐案累積；用於優化殘留誤差 |

### 使用順序：先粗後細，不並用

**正確流程**：
1. 用粗濾網（流程規範）處理原始逐字稿 → 輸出初版步驟草稿
2. 遇到仍有疑問的辨識 → 查細濾網（Debug 紀錄）核對已知錯誤模式
3. Phase 4 使用者確認後 → 把新發現的錯誤模式補進細濾網文件

**避免的錯誤做法**：
- 一開始就同時參照兩份文件 → 輸出結果因為混合規則而品質不一
- 把 Debug 紀錄的條目直接嵌入流程規範 → 兩份文件的職責混淆，工具變臃腫

### 為什麼要分開放？

粗濾網（流程規範）的目標是**通用性**：不管影片是 PS 還是 Illustrator，這套規範都適用。
細濾網（Debug 紀錄）的目標是**精確性**：針對已知的具體錯誤模式，給出確定的修正對照。

把兩者混在一起，規範會越來越長、越來越具體，反而無法「通用」，讓 AI 也難以正確套用。

## 上下游流程設計

### 資訊的整體流向

```
[影片教學]
    ↓ MEMO.AI 語音轉錄
[原始逐字稿]（含大量雜訊）
    ↓ 粗濾網（Phase 3 流程規範）
    ↓ 細濾網（Phase 3 Debug 紀錄，二次優化）
[工具規格草稿]（結構化步驟 + 可調參數）
    ↓ 使用者實機確認（Phase 4）
[定稿規格草稿]
    ↓ 下游接手依用途轉換
[各目標格式] 例如：PS 工具 Python 程式碼 / 未來的 Illustrator 工具
```

### 上游的責任（本流程所在位置）

上游（影片 → 規格草稿）的責任是：
1. **保留下游需要的資訊**：即使目前沒有對應的下游，也要保留潛在有用的資訊
2. **標注可調參數的作用**：讓下游知道哪些參數可以程式化、哪些需要使用者介入
3. **標注自動化程度（A/B/C）**：讓下游能直接套用，不需重新判斷
4. **保留原始語音辨識文字**：供 Debug 溯源，讓下游能理解資訊的原始形態

### 下游的責任（其他規範文件）

下游（各目標格式的開發規範）的責任是：
1. **明確說明從上游需要什麼格式的資訊**（在自己的規範裡寫清楚）
2. **當上游規範更新時，確認自己的需求是否仍被滿足**
3. **把使用上游資訊時發現的格式問題，回饋給上游規範**

### 雙向維護原則

上下游規範是**雙向耦合**的，任何單向更新都可能造成資訊斷層：

- 更新上游規範 → 確認下游規範的需求是否仍被滿足
- 更新下游規範 → 確認上游是否有提供所需的資訊格式
- 每次更新規範文件時，習慣性往另一個方向看一遍

## 對未來 AI 的工作建議

### 遇到不確定的決策時

1. 先查粗濾網文件（流程規範）：是否已有明確規則？
2. 再查細濾網文件（Debug 紀錄）：是否是已知的語音辨識錯誤模式？
3. 都沒有答案時 → 向使用者提問，把結論補進對應文件

### 遇到需要更新規範的情況時

1. 確認這個更新屬於哪個層次：
   - 通用流程規則 → 更新粗濾網（流程規範）
   - 具體語音辨識錯誤 → 更新細濾網（Debug 紀錄）
   - 概念或設計理念的改變 → 更新本文件
2. 更新後，往上下游各看一遍，確認一致性
3. 更新 MENU.md

### 遇到使用者說出設計決策時

如果使用者說的話讓你覺得「這是一個概念性的設計決策，不只是這次任務的指示」，考慮更新本文件。
這份文件是活的，會隨著工作方式的演進持續更新。
