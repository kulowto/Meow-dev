# (AI_Read) AI 協作行為規範（PS 工具補充）

> ⚠️ **請先閱讀全域規範：**
> `架構Wiki與AI的協作環境/(AI_Read) (Global Prompt) AI開發協作行為規範.md`
>
> 本文件是 PS 工具開發的**補充說明**，包含 PS 特有的範例與細節。
> 全域四條規則同樣適用，本文件不重複列出，只補充 PS 情境的具體作法。

---

## 規則一：查閱再實作，不要憑空猜測（禁止幻覺）

**問題描述：**
前任 AI 在實作 PS API 相關功能時，直接猜測函式的用法與參數，沒有先查閱現有程式碼或參考文件，導致產出的程式碼出現大量錯誤，每次執行都需要反覆修正。

**規範：**
- 實作任何 PS API 功能前，先查閱 `ps_utils.py` 是否已有對應函式
- 若需要使用 Action Descriptor / charID / stringID，先查閱現有已知可行的寫法，再照格式套用
- 不確定某個 API 是否可行時，**先做最小單元測試**（見規則三），不要直接整合進流程
- 確實不確定時，**直接告訴使用者**，討論後再決定方向，不要自行假設

**參考：**
- 已知技術問題：`(AI_Read) Photoshop 自動化腳本開發.md` §已知技術問題（PS 27.6）
- 低階函式清單：直接查閱 `ps_tools/ps_utils.py` 的函式定義

---

## 規則二：跳過功能時必須留下紀錄

**問題描述：**
前任 AI 在執行某個功能到一半時，為了先處理其他事情而靜默跳過原本的任務，既沒有告知使用者，也沒有留下任何 TODO 或說明，導致使用者事後才發現功能沒有完成，必須重新追查脈絡。

**規範：**
- 若因優先級考量決定先跳過某功能，必須在跳過前告知使用者並說明原因
- 在程式碼中留下清楚的 `# TODO:` 標記，說明：跳過了什麼、預計怎麼做、為什麼先跳過
- 在本次對話或相關文件中留下書面紀錄，確保接手的人（AI 或人）能快速了解狀況

**格式範例：**
```python
# TODO: chr_halfTone 封裝
# 功能：去色 → Levels → Color Halftone → Multiply → 移入 chr_effects
# 跳過原因：優先確認 Phase 1 selectSubject 方案的實機執行結果
# 參考：(AI_Read) Photoshop 自動化腳本開發.md §Phase 3 > chr_halfTone
```

## 規則三：先單元測試，再整合流程

**問題描述：**
前任 AI 習慣一次性修改大量程式碼後直接整合進主流程，結果執行後出現一大堆錯誤，難以定位問題來源，反覆除錯消耗大量 Token，效率極低。

**規範：**
- 對任何不確定能否執行的函式或 API，先在獨立的 Playground 腳本中做**最小單元測試**
- 確認該函式可以正確執行後，才將其整合進 `ps_auto.py` 或工具模組
- Playground 腳本放在 `ps_tools/playground/` 目錄下，每個測試功能一個獨立檔案
- 命名格式：`test_[功能名稱].py`，例如 `test_color_halftone.py`

**測試流程：**
1. 建立 `test_xxx.py`，只測試該函式，不執行其他流程
2. 確認 Photoshop 已開啟且目標文件已載入
3. 執行測試，確認無錯誤且視覺結果正確
4. 通過後，才將程式碼移入正式工具模組

**判斷原則：**
- 如果測試通過但整合後出錯 → 問題在**流程組合**，縮小範圍除錯
- 如果測試本身就出錯 → 問題在**函式實作**，先修函式再談整合

## 規則四：先對齊思路，再動手實作

**問題描述：**
前任 AI 收到任務後直接開始實作，沒有先確認自己的理解是否與使用者的意圖一致，結果方向做錯，白費功夫。

**規範：**
- 開始任何實作前，先用**一到三句話**說明你打算怎麼做、為什麼這樣做
- 等使用者明確確認（「對」、「可以」、「照這樣做」）後，才開始動手
- 若有多種可行方案，提出選項讓使用者選擇，不要自行決定

**格式範例（AI 開始實作前應說的話）：**
> 「我的計畫是：先在 `test_halftone.py` 寫一個最小測試，確認 `_color_halftone()` 能正確執行，然後再封裝成 `tools/halftone_shadow.py`，最後補工具文件。這樣可以嗎？」

## 總結對照表

| 規則 | 禁止行為 | 正確做法 |
|------|---------|---------|
| 1. 查閱再實作 | 猜測 API 用法直接寫 | 先查 ps_utils.py 或現有範例，不確定就問 |
| 2. 跳過要留紀錄 | 靜默跳過任務 | 告知使用者 + 留 TODO 標記 |
| 3. 先單元測試 | 大批改動後直接整合 | Playground 測通後再整合 |
| 4. 先對齊思路 | 直接開始實作 | 說明計畫 → 等確認 → 才動手 |
